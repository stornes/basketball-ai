# HoopsVision v1.7.0 — Observational Stats Manifesto

> **The pipeline IS the scorekeeper. It must observe, not estimate.**

This document defines the root causes, fixes, and implementation roadmap for
three broken observational stats: **FGA** (field goal attempts), **MIN** (playing
time), and **AST** (assists). All three must be derived from video observation —
no estimation, no API dependency for core detection.

---

## 1. Principles

These principles apply to all observational stat development in HoopsVision:

1. **Observe, don't estimate.** Playing time comes from tracking player presence
   on court, not dividing game minutes by roster size. FGA comes from detecting
   actual shot attempts, not inferring from score totals.

2. **Resolution-independent thresholds.** Every pixel-based threshold must scale
   with frame dimensions. A detector tuned at 1080p must work at 4K without
   manual adjustment.

3. **Identity before aggregation.** Stats are meaningless without knowing WHO
   performed the action. Jersey resolution is a prerequisite, not a nice-to-have.

4. **Layered confidence over binary detection.** Prefer confidence scores that
   can be thresholded over hard yes/no decisions. Log sub-threshold candidates
   for future training data.

5. **API is truth for what it covers.** When kamper.basket.no provides a stat
   (points, fouls, FTM/FTA, 2P made, 3P made, period scores), that IS the
   value — not a validation source. Pipeline detection of those same stats
   serves as fallback and cross-check. See §2.5.

6. **Growth-mindset assessment.** Following `coaching.yaml`, all player
   evaluations use development labels (Impact Player / Solid Foundation /
   Developing / Growth Opportunity), never letter grades. Stats exist to inform
   coaching insights, not to rank children.

- ✅ 2P FGM and 3P FGM are truth
- ❌ 2P FGA and 3P FGA are NOT truth — they equal FGM
- ✅ 1P FTA IS truth — free throw attempts are genuinely tracked (11/26 = 42%)

- ❌ **FGA** — 2P and 3P attempts. Only makes (denominator = numerator)
- ❌ **FG%** — cannot compute without real attempts
- ❌ **MIN** — column exists but 00:00 (not populated)
- ❌ **AST** — no column at all
- ❌ **REB** — OR/DR columns exist but all 0 (not populated)
- ❌ **TOV** — TOR column exists but all 0 (not populated)
- ❌ **STL, BLK** — no columns at all

**Implication for the pipeline:** FGA must come entirely from video detection.
The API's 2P/3P "attempt" numbers are useless (always = made). The pipeline's
FGA is the ONLY source of real attempt data, making the three-layer shot
detection fix (§4.1) critical.

Points, fouls, free throws (made AND attempted), and made field goals from the
API are truth. The pipeline must use API values for these stats when available.
Pipeline detection serves as fallback when the API is unavailable.

## 3. Root Cause Analysis

### 3.1 FGA: Shot Over-Detection

**File:** `app/events/shot_detector.py`

The shot detector uses ball trajectory arc heuristics. Three compounding problems:

**Problem A: Resolution-blind thresholds**

```python
MIN_VERTICAL_DISPLACEMENT = 40   # pixels — at 4K this is 1.85% of frame
COOLDOWN_FRAMES = 30             # at sample_rate=3, this is 3.2 seconds
```

At 4K (3840×2160), 40 pixels of vertical movement is a bounce pass, a
dribble, or a rebound arc. At the resolution the detector was likely developed
(1080p), 40px = 3.7% of frame height — twice the relative threshold.

**Problem B: No spatial filtering**

The detector fires on ANY ball arc anywhere on the court. Cross-court passes,
warm-up tosses, and player movements near the ball all trigger false positives.
There is no requirement that the ball be near a basket.

**Problem C: Dual-trigger false positives**

`_check_arc()` and `_check_shot_on_ball_loss()` can both fire on the same
physical event within the cooldown window, especially when the ball is
temporarily occluded by a player during a real shot.

### 3.2 MIN: Wrong Key Lookup

**File:** `app/analytics/box_score.py`, lines 497-506

```python
# CURRENT CODE (broken):
track_frame_count: dict[int, int] = {}
for t in tracks:
    track_frame_count[t.track_id] = track_frame_count.get(t.track_id, 0) + 1

for key, p in players.items():
    count = track_frame_count.get(p.player_id, 0)
    # p.player_id = jersey number (canonical, e.g., 4)
    # track_frame_count keys = raw tracker IDs (e.g., 2847, 3102, 5190)
    # These never match → count = 0 → min_played = 0
```

The `_get_player()` function (line 400) correctly merges fragmented tracks by
using jersey number as the canonical key. But the minutes calculation bypasses
this merge — it looks up `p.player_id` (jersey number) directly in
`track_frame_count` which is keyed by raw `track_id`.

**Secondary problem:** `_track_to_key` only maps tracks that appear in shot
events (448 of 5775 tracks). For MIN, we need frame counts from ALL tracks
that resolved to a jersey number, not just shooters.

### 3.3 AST: Possession Tracker Broken at Every Level

**File:** `app/events/possession.py`

Three independent failures compound to produce 4 possessions instead of ~150:

**Failure 1: Random team assignment (line 96-99)**
```python
@staticmethod
def _assign_team(track_id: int) -> str:
    return "team_a" if track_id % 2 == 0 else "team_b"
```

This assigns teams based on track ID parity — completely random. Yet Stage 3.5
of the pipeline already classifies 5771 tracks into home (3692) and away (2079)
using jersey colour. This classification is never passed to PossessionTracker.

**Failure 2: Tight proximity threshold (line 13)**
```python
POSSESSION_DISTANCE_PX = 80  # max pixels between ball and player
```

At 4K, 80px = 2.1% of frame width. During a dribble, the ball can be 100-150px
from the player's bbox centre. This threshold rejects most real possessions.

**Failure 3: High minimum duration (line 14)**
```python
MIN_POSSESSION_FRAMES = 5  # at sample_rate=3, this is 15 real frames = 0.5s
```

A catch-and-shoot takes ~0.3 seconds. A pass-and-drive ~0.4 seconds. At 5
sampled frames minimum, most real possessions are filtered out.

**Failure 4: Sequencing problem (architectural)**

PossessionTracker runs DURING frame-by-frame detection (Stage 3). Team
classification runs AFTER detection (Stage 3.5). The tracker can't know team
assignments because they don't exist yet when it processes each frame.

### 3.4 Shot Chart: Broken Homography Coordinates

**File:** `app/vision/court_mapper.py`

The court mapper uses Hough line detection to find court boundaries, then
computes a homography from pixel space to court feet (94 × 50 ft full court).

**Problem A: Unreliable calibration**

`cv2.HoughLinesP` finds edges in the first 30 frames. In a typical gym
environment, it picks up scoreboard edges, bleacher lines, wall markings,
and ad boards alongside actual court lines. The "outermost" lines are often
NOT the court boundary → the resulting homography maps to the wrong rectangle.

**Problem B: Full-court → half-court mismatch**

`CourtMapper` maps to full court (94 ft), but `ShotChartGenerator` renders
half court (47 ft). Shots at the far basket (y=47-94) need to be mirrored
to the near-basket frame of reference: `y_chart = 94 - y`, `x_chart = 50 - x`.
This mirroring is not implemented — far-basket shots either fall off the chart
or plot at wrong positions.

**Problem C: Ball position at arc apex, not shot origin**

`shot.court_position` is set from `ball.bbox.center` at the frame when the
shot is detected. During an arc, the ball is airborne — its pixel position is
above the court plane. The homography assumes all points are ON the court
surface. An airborne ball at y=500px (above the basket) maps to a completely
wrong court position because the homography's perspective projection only
works for ground-plane points.

**Evidence:** 331 shots with coordinates show Y mean of 44.0 ft (midcourt).
Only 7% are within 15 ft of a basket. Expected: ~65-70% within 15 ft.

## 4. Proposed Fixes

### 4.1 FGA Fix: Three-Layer Defence

**Layer 1: Resolution-aware thresholds** (1 hour)

```python
# shot_detector.py — __init__
self.MIN_VERTICAL_DISPLACEMENT = int(frame_height * 0.04)  # 86px at 4K
self.COOLDOWN_FRAMES = int(fps * 5.0 / sample_rate)        # 5s real cooldown

# Existing constants become class-level defaults for 1080p compatibility:
MIN_VERTICAL_DISPLACEMENT_FALLBACK = 40
COOLDOWN_FRAMES_FALLBACK = 30
```

Expected reduction: 478 → ~250 (eliminates small-arc false positives)

**Layer 2: Basket proximity gate** (2 hours)

Add a spatial filter: only trigger shot detection when the ball is within a
"shot zone" — defined as 25% of frame width from the nearest detected basket.

```python
def _in_shot_zone(self, ball_x: float) -> bool:
    """Check if ball is near enough to a basket to be a shot."""
    if self._last_basket_detection is None:
        return True  # Can't filter without basket position
    basket_x = self._last_basket_detection.bbox.center[0]
    max_dist = self.frame_width * 0.25
    return abs(ball_x - basket_x) < max_dist
```

Add this check at the top of `update()` before arc analysis.

Expected reduction: 250 → ~140 (eliminates mid-court false positives)

**Layer 3: Temporal dedup** (1 hour)

Within any 4-second window, emit at most 1 shot event per basket. If both
`_check_arc()` and `_check_shot_on_ball_loss()` fire within the window, keep
the event with the higher basket proximity score.

```python
# After creating a shot event, check recent history:
if self._recent_shots and (event.timestamp_sec - self._recent_shots[-1].timestamp_sec) < 4.0:
    # Dedup: keep the one closer to basket
    if self._score_proximity(event) > self._score_proximity(self._recent_shots[-1]):
        self._recent_shots[-1] = event
    return None  # Suppress duplicate
```

Expected reduction: 140 → ~120-130 (eliminates remaining arc+ball-loss doubles)

### 4.2 MIN Fix: Aggregate Tracks Per Jersey

**Phase 1: Fix key lookup using existing track_to_key** (1 hour)

```python
# box_score.py — replace lines 497-506

# Aggregate frame counts per canonical player key (not raw track_id)
jersey_frame_count: dict[tuple[str, int], int] = {}
for t in tracks:
    team = t.team or "unknown"
    raw_key = (team, t.track_id)
    canonical = _track_to_key.get(raw_key, raw_key)
    jersey_frame_count[canonical] = jersey_frame_count.get(canonical, 0) + 1

for key, p in players.items():
    count = jersey_frame_count.get(key, 0)
    if count > 0 and fps > 0:
        # CRITICAL: multiply by sample_rate — we only process every Nth frame
        p.min_played = (count * sample_rate) / fps / 60.0
        p.stat_sources.setdefault("min_played", StatSource.HEURISTIC)
```

**Limitation:** `_track_to_key` only maps shooter tracks (448 of 5775). Players
who never shoot will show MIN=0. This is acceptable for v1.7.0 — the column
appears for shooters, and a note in the report explains the limitation.

**Phase 2: Extended jersey resolution (future v1.8.0)** (4 hours)

Resolve jersey numbers for ALL tracks with ≥20 frames (significant on-court
presence), not just shooter tracks. Use VLM for high-frame tracks, Sherlock
for the rest.

Estimated cost: ~5775 tracks but ~90% have <20 frames. ~500-600 meaningful
tracks × $0.01/crop = ~$6 per game. Acceptable.

### 4.3 AST Fix: Two-Pass Possession Tracking

**The key architectural insight:** Possessions cannot be tracked in real-time
because team identity isn't known until after Stage 3.5. Solution: store raw
ball-player proximity data during Stage 3, then run possession segmentation
as a post-processing step after team classification.

**Step 1: Store ball-player proximity data during detection** (30 min)

During the frame-processing loop, record `(frame_idx, ball_position,
closest_player_track_id, distance)` tuples. This is lightweight — one tuple
per frame, ~52K entries for a full game.

**Step 2: Wire team classification to PossessionTracker** (30 min)

```python
# possession.py — modify __init__
def __init__(self, fps: float, frame_width: int = 1920,
             team_map: dict[int, str] | None = None):
    self.fps = fps
    self.team_map = team_map or {}
    self.POSSESSION_DISTANCE_PX = int(frame_width * 0.05)  # 192px at 4K
    self.MIN_POSSESSION_FRAMES = 2  # 0.2s at sample_rate=3

def _assign_team(self, track_id: int) -> str:
    return self.team_map.get(track_id, "unknown")
```

**Step 3: Run PossessionTracker in post-processing** (1 hour)

After Stage 3.5 (team classification), replay the stored ball-player proximity
data through PossessionTracker with the real team map.

```python
# run_analysis.py — after Stage 3.5
print("Stage 3.5b: Post-processing possession tracking...")
team_map = {t.track_id: t.team for t in classified_tracks}
post_poss_tracker = PossessionTracker(
    fps=meta.fps, frame_width=meta.width, team_map=team_map
)
for bp in ball_player_proximity_data:
    post_poss_tracker.update(bp.ball, bp.players, bp.frame_idx)

# Replace the original (broken) possession events
result.possession_events = post_poss_tracker.events
```

**Step 4: Verify assist detection works** (30 min)

The `AssistDetector` code is already correct. Once PossessionTracker produces
~150 events with real team assignments, assists should flow automatically:
- Made shot by Player B
- Same-team possession by Player A within 6 seconds before the shot
- A ≠ B → assist

Expected AST: 15-25 per game (typical U16 rate: ~12-15 assists per team).

### 4.4 Shot Chart Fix: Basket-Relative Coordinates

The broken homography (§3.4) cannot be fixed reliably with Hough lines alone —
gym environments have too many non-court edges. Instead, use the **basket
detections** the model already produces (class_id=1) as anchor points.

**Approach: Basket-relative shot positioning** (2 hours)

```python
# shot_detector.py — when a shot is detected, record which basket it targets
# The nearest basket detection in recent frames = target basket

def _assign_shot_court_position(
    self, shot: ShotEvent, basket_det, frame_width: int, frame_height: int
) -> tuple[float, float]:
    """Map shot to half-court coords using basket position as anchor.

    Basket center pixel → fixed court position (25, 5.25) ft.
    Ball pixel position → offset from basket in court feet.
    Scale factor: court_width / frame_width for horizontal,
                  approximate using known basket-to-3pt-line distance.
    """
    basket_px = basket_det.bbox.center  # (bx, by) in pixels
    ball_px = (shot.ball_x, shot.ball_y)  # ball at shot detection

    # Pixels per foot estimated from basket size
    # A regulation basket is 18 inches = 1.5 ft diameter
    # The detected basket bbox width in pixels ÷ 1.5 = px/ft
    basket_width_px = basket_det.bbox.width
    px_per_ft = basket_width_px / 1.5 if basket_width_px > 0 else frame_width / 50

    # Ball offset from basket in feet
    dx_ft = (ball_px[0] - basket_px[0]) / px_per_ft
    dy_ft = (basket_px[1] - ball_px[1]) / px_per_ft  # invert Y (pixel Y increases downward)

    # Map to half-court: basket at (25, 5.25)
    court_x = 25.0 + dx_ft
    court_y = 5.25 + dy_ft

    return (court_x, court_y)
```

**Why this works better:**
- No homography needed — uses the model's own basket detection as reference
- Resolution-independent — basket size in pixels provides the scale factor
- Works for both baskets — shot detector already knows which basket is targeted
- Airborne ball issue mitigated — we only need the horizontal offset from basket
  (vertical offset in pixels maps to court depth via perspective, but for a
  half-court chart, approximate accuracy is sufficient)

**Limitation:** Perspective distortion means shots at the far basket have less
accurate positioning (smaller basket bbox, more foreshortening). Accept ±3 ft
accuracy for far-basket shots.

### 4.5 Observational Honesty: What AST Actually Measures

> **Principle 1 applies here: Observe, don't estimate.**

The proposed AST metric is **not** observational in the same way FGA and MIN are.

- **FGA** observes a physical event: ball trajectory arc near a basket. The camera
  sees the shot happen.
- **MIN** observes player presence: tracked bounding boxes on court across frames.
  The camera sees the player.
- **AST** infers a relationship from **sequential ball proximity**: Player A had the
  ball near them, then Player B scored within 6 seconds. The camera never sees
  a pass — it sees two independent proximity events and assumes causation.

**What this misses:**

| Scenario | Real Assist? | Detected? | Why Not |
|----------|-------------|-----------|---------|
| Skip pass (A → C, ball airborne 1.5s) | Yes | ❌ | Ball not near anyone during transit |
| Hockey assist (A → B → C scores) | A deserves secondary | ❌ | Only last possession before shot counted |
| Kick-out (drive → pass to corner) | Yes | ⚠️ Maybe | Only if ball-player distance < threshold during catch |
| Dribble hand-off | Yes | ✅ Likely | Players close together, proximity transfers cleanly |
| Post entry pass | Yes | ⚠️ Maybe | Depends on distance threshold at 4K resolution |

**Honest classification:**

The AST metric has `StatSource.HEURISTIC` provenance — it is an inference, not an
observation. This is fundamentally different from FGA (observed arc) and MIN
(observed presence). The manifesto acknowledges this distinction rather than
presenting AST as equivalent observational data.

**True observational AST — IMPLEMENTED (v1.7.0):**

The proximity-based AST proxy has been replaced with a two-tier observational
system. Three new modules provide true pass detection:

1. **Ball-in-hands detector** (`ball_possession.py`) — classifies whether a
   specific player is holding/dribbling the ball via bbox overlap. The ball
   center must fall inside the player's upper-body region (top 60% height,
   inner 60% width). This is a direct observation — the camera sees the ball
   in the player's hands. Requires 2 consecutive frames to confirm possession,
   filtering noise. Emits `PossessionTransition` events on holder changes.

2. **Pass trajectory tracking** (`pass_detector.py`) — detects passes by
   tracking ball movement between two player possessions. A pass is observed
   when: ball leaves Player A's bbox → ball travels through open space → ball
   enters Player B's bbox within 2 seconds, covering > 5% of frame width.
   The ball centroid path is stored as trajectory data on the PassEvent.
   Pass type classified from trajectory shape: lob (high arc), bounce (low
   dip), chest (flat). All thresholds resolution-scaled via frame_width.

3. **Two-tier assist attribution** (`assist_detector.py`) — rewritten with:
   - **Tier 1 (observational):** Search PassEvents for a pass TO the scorer
     within 6 seconds. The camera saw the pass → this IS the assist.
     Source: `"pass"`.
   - **Tier 2 (heuristic fallback):** When no PassEvents match, fall back to
     proximity-based possession matching (same as v1.6.x). Source: `"proximity"`.
   - Every `AssistEvent` carries a `source` field indicating which tier
     produced it — honest provenance for downstream consumers.

**Pipeline integration (Stage 3 + Stage 3.5c):**
- `BallInHandsDetector` and `PassDetector` run per-frame during Stage 3
  alongside shot/possession detection
- Pass events labeled with team info after Stage 3.5 classification
- `AssistDetector.check()` receives observed passes in Stage 3.5c

**Test coverage:** 19 unit tests covering ball-in-hands detection, pass
detection (valid/rejected/trajectory/classification), and two-tier assist
attribution (pass-based, proximity fallback, team filtering, time windows).
306 total tests pass (287 existing + 19 new).

**Remaining limitations:**
- Passes where the ball is fully occluded (e.g., behind a player's body on
  a bounce pass) will be missed. The ball tracker loses the ball during
  occlusion. This is an honest observational gap, not an estimation shortcut.
- VLM fallback for contested/ambiguous ball possession deferred to v1.8.0.
  The bbox overlap heuristic handles clear possessions well but may miss
  ~20% of contested-ball situations.
- Skip passes (A → C with ball airborne >2s) exceed the transit window and
  will be caught by Tier 2 proximity fallback instead of Tier 1 observation.

## 8. Anti-Patterns (What We Will NOT Do)

1. **No estimation of stats.** We will not estimate FGA from score totals, MIN
   from roster size, or AST from team passing patterns. Every stat must be
   directly observed from video.

2. **API is truth, pipeline is fallback.** For stats the API covers (points,
   fouls, FT, 2P made, 3P made), API values ARE the data. The pipeline must
   still detect these stats independently so it works when the API is
   unavailable — but when API data exists, it takes precedence.

3. **No magic numbers.** Every threshold must be documented with its derivation
   and must scale with resolution. No `80` or `40` without a comment explaining
   the physical meaning.

4. **No architectural shortcuts.** The possession tracker's team assignment must
   use real data. We will not "fix" it by hardcoding team heuristics or using
   court-half as a proxy.

5. **No false equivalence between metrics.** AST has two tiers: Tier 1
   (pass-observed) is true observation like FGA; Tier 2 (proximity-inferred) is
   heuristic like early MIN. Every `AssistEvent.source` field marks which tier
   produced it. Reports must distinguish observed vs inferred assists. See §4.5.

### VLM Model: gemini-3.1-pro-preview → gemini-3-flash

**Problem:** `gemini-3.1-pro-preview` has a hard 25 RPM rate limit on Google AI Studio,
